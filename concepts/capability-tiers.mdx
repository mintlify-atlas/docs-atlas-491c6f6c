Every feature in Fullmag is tagged with exactly one of three capability tiers. The same tag appears consistently in Python errors, CLI output, API responses, browser badges, and documentation. No feature is "sort of supported."

## Three tiers

<CardGroup cols={3}>
  <Card title="semantic-only" icon="tag">
    Legal in the Python API and `ProblemIR`. Can be serialized, validated, and planned. Not numerically implemented. Attempting to run will produce a clear capability error.
  </Card>
  <Card title="internal-reference" icon="flask">
    Numerically implemented inside `fullmag-engine` or an equivalent crate, but not wired to the public `Simulation.run()` path. Available for internal calibration and testing only.
  </Card>
  <Card title="public-executable" icon="circle-check">
    Fully wired end-to-end: Python `Simulation.run()` → plan → runner → engine → artifacts. This is the only tier you can rely on in production scripts.
  </Card>
</CardGroup>

## What each tier means in practice

<AccordionGroup>
  <Accordion title="semantic-only">
    A `semantic-only` feature is part of the public Python API and the `ProblemIR` schema. You can write it in your script, and it will serialize and validate without error. However, the planner will refuse to produce an executable plan that uses it.

    Use `semantic-only` features to describe the physical model you intend to simulate. When the feature becomes executable, your script will work without modification.

    **Example**: `fm.Cylinder(...)` can be placed in a script today. The planner will report that `Cylinder` geometry requires an active-mask voxelizer that is not yet wired.
  </Accordion>
  <Accordion title="internal-reference">
    An `internal-reference` feature has a working numerical implementation but it is not exposed through the public `Simulation.run()` path. It may be used by the development team for calibration, benchmarking, or testing.

    **Example**: `Magnetoelastic` has a small-strain implementation with prescribed-strain H_mel wired into H_eff, but it is not accessible from a public `Simulation.run()` call.
  </Accordion>
  <Accordion title="public-executable">
    A `public-executable` feature is fully wired from Python authoring through planning and runner dispatch to artifact output. You can use it in production scripts with confidence.

    Not all backends are equally covered. The tier table below specifies which combination of feature + backend + precision is actually executable.
  </Accordion>
</AccordionGroup>

## Checking capability before running

`Simulation.plan()` returns the capability status for your problem without starting execution:

```python
import fullmag as fm

geom = fm.Box(size=(200e-9, 20e-9, 5e-9), name="strip")
mat  = fm.Material(name="Py", Ms=800e3, A=13e-12, alpha=0.02)
mag  = fm.Ferromagnet(name="strip", geometry=geom, material=mat, m0=fm.init.uniform([1, 0, 0]))

problem = fm.Problem(
    name="capability_check",
    magnets=[mag],
    energy=[fm.Exchange(), fm.Demag()],
    study=fm.Relaxation(algorithm="llg_overdamped", max_steps=5000),
    discretization=fm.DiscretizationHints(fdm=fm.FDM(cell=(2e-9, 2e-9, 2e-9))),
)

plan = fm.Simulation(problem, backend="fdm", precision="double").plan()

print(plan.capability_status)   # "public-executable" | "semantic-only" | "internal-reference"
print(plan.chosen_backend)      # resolved backend
print(plan.diagnostics)         # human-readable explanation
```

If any feature in the problem is `semantic-only`, `plan()` will report which feature blocked execution and why.

## Notes on specific features

<Warning>
The following features are currently `semantic-only`. Including them in a problem will produce a capability error at planning time:

- **`Cylinder` geometry** — Curved-boundary FDM execution requires an active-mask voxelizer that is not yet wired to the public execution path.
- **`Hybrid` backend** — Coupled FDM/FEM planning is not yet implemented. `backend="hybrid"` with `mode="hybrid"` is legal in Python and `ProblemIR` but will not produce an executable plan.
- **`InterfacialDMI`** — No numerical implementation exists. The term can be included in the energy list and will serialize, validate, and appear in `ProblemIR`, but execution will be blocked by the planner.
</Warning>

<Note>
**CUDA single precision status**: `precision="single"` is `public-executable` on CUDA FDM. The CPU reference FDM runner is `double`-only. GPU `double` parity is required before GPU `single` is considered stable; use `double` as the trusted reference when calibrating results.
</Note>

## Capability tags in practice

The same tier tag appears everywhere the user encounters a feature:

- **Python errors**: `CapabilityError: 'Cylinder' geometry is semantic-only. Requires active-mask voxelizer.`
- **CLI output**: the `fullmag plan` command prints a capability summary table.
- **API responses**: `/v1/sessions/:id` includes `capability_status` in the plan summary.
- **Browser badges**: the run page shows tier badges next to each feature in the problem summary.
- **Documentation**: every feature page states its current tier.

<Tip>
Run `fullmag plan script.py` before submitting long jobs. The planner will report the capability status, resolved backend, and any blocking features without starting execution.
</Tip>

## Auto-backend resolution

When `backend="auto"`, the planner applies deterministic resolution rules:

- For `mode="strict"` and `mode="extended"`: resolves to `fdm` during bootstrap planning.
- For `mode="hybrid"`: raises a planning error. Hybrid requires an explicit `backend="hybrid"`.
- Hybrid planning is a deliberate opt-in, never a fallback from `auto`.

The resolved backend and the reason for the resolution are always recorded in the `ExecutionPlanIR` and the artifact `execution_plan.json`.
